# 💬 Mini Chat на C + FastCGI

Простой веб-чат, реализованный на языке C с использованием FastCGI и nginx.
Проект создан в учебных целях — чтобы показать, **как работали веб-приложения в 90-е годы и как эволюционировали серверные технологии**.

---

# 📜 История CGI

В начале развития интернета (90-е годы) веб-сайты были в основном статическими — обычные HTML-файлы без какой-либо логики.

Чтобы добавить динамику, появился **CGI (Common Gateway Interface)** — стандарт взаимодействия веб-сервера с внешними программами.

### 🔧 Как работал CGI:

1. Пользователь отправляет запрос (например, форма)
2. Веб-сервер:

   * запускает отдельную программу (например, на C, Perl)
   * передаёт ей данные (через переменные окружения / stdin)
3. Программа выполняется
4. Возвращает HTML
5. Процесс **завершается**

---

## ❗ Проблема CGI

Главный минус:

> ❌ **каждый запрос = запуск нового процесса**

Это означало:

* огромные затраты ресурсов
* медленную работу при нагрузке
* плохую масштабируемость

Если у тебя 100 пользователей → 100 процессов.

---

# ⚡ Появление FastCGI

Чтобы решить проблему CGI, был создан **FastCGI**.

### 💡 Идея FastCGI:

> программа запускается **один раз** и остаётся работать

---

## 🔄 Как работает FastCGI:

1. Программа запускается заранее
2. Она **ожидает запросы в цикле**
3. Веб-сервер передаёт ей данные через сокет
4. Программа обрабатывает запрос
5. Возвращает ответ
6. **НЕ завершается**, а ждёт следующий запрос

---

## 🚀 Преимущества FastCGI

* не создаёт процесс каждый раз
* быстрее CGI
* меньше нагрузка на систему
* можно обрабатывать много запросов подряд

---

# 🧬 Наследие CGI

Несмотря на то, что CGI сейчас почти не используется, его идеи живы до сих пор:

* передача данных через HTTP
* разделение сервера и логики
* обработка запросов программой

Современные технологии — это эволюция CGI:

* PHP (изначально работал как CGI)
* Python (WSGI)
* Node.js
* Go HTTP servers

Все они решают ту же задачу:

> “как обработать HTTP-запрос и вернуть ответ”

---

# 🤔 Почему выбран FastCGI

В этом проекте используется FastCGI, потому что:

> программа запускается **один раз и работает постоянно**,
> а не создаётся заново при каждом запросе, как в CGI.

Это позволяет:

* уменьшить нагрузку
* ускорить обработку запросов
* приблизиться к современным backend-подходам

# ⚙️ Сборка и запуск

### Установка зависимостей:

```bash
sudo apt update
sudo apt install gcc libfcgi-dev nginx spawn-fcgi
```

### Компиляция:

```bash
gcc chat.c -o chat.fcgi -lfcgi
```

### Запуск FastCGI:

```bash
spawn-fcgi -p 9090 ./chat.fcgi
```

---

# 🌐 Настройка nginx

Пример конфигурации:

```nginx
server {
    listen 9990;

    location / {
        include fastcgi_params;
        fastcgi_pass 127.0.0.1:9090;
        fastcgi_param SCRIPT_FILENAME /path/to/chat.fcgi;
    }
}
```

Запуск:

```bash
nginx -t
nginx
```
